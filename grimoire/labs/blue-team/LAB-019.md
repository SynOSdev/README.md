## 📖 Background

Incident Response (IR) is the organized approach to handling security breaches and attacks. The NIST framework defines six phases that form a cycle, not a one-time process. Effective IR requires preparation before incidents occur, clear procedures during response, and thorough review afterward.

Key concepts:
- **Preparation** — Building IR capabilities, training teams, establishing contacts
- **Identification** — Detecting and confirming that an incident has occurred
- **Containment** — Limiting the scope and impact of the incident
- **Eradication** — Removing the root cause and threat actor access
- **Recovery** — Restoring systems to normal operations
- **Lessons Learned** — Documenting findings and improving future response

## 🔬 Lab Instructions

### Phase 1: Preparation and Playbook Development

1. Create an IR documentation structure:
   ```bash
   mkdir -p ~/grimoire/ir/{playbooks,evidence,timelines,reports}
   ```
2. Define your incident classification severity levels:
   - **SEV-1** — Critical: Active data exfiltration, ransomware spreading
   - **SEV-2** — High: Confirmed unauthorized access, malware detected
   - **SEV-3** — Medium: Suspicious activity, policy violation
   - **SEV-4** — Low: Reconnaissance detected, failed attacks
3. Build a playbook template for a phishing incident:
   - Identification triggers (user report, email filter alert, IDS alert)
   - Initial triage steps and severity classification
   - Containment actions (isolate host, block sender, disable account)
   - Eradication steps (remove malware, reset credentials)
   - Recovery verification checklist
4. Define escalation paths and communication templates
5. Record your terminal session for documentation: `script -a ~/grimoire/ir/session.log`

### Phase 2: Evidence Preservation

1. Practice forensic image acquisition:
   ```bash
   # Create a test disk image (use a small file for the lab)
   dd if=/dev/zero of=/tmp/test_disk.img bs=1M count=100
   mkfs.ext4 /tmp/test_disk.img
   # Create a forensic copy with hash verification
   dd if=/tmp/test_disk.img of=/tmp/evidence.img bs=4096
   md5sum /tmp/test_disk.img /tmp/evidence.img
   sha256sum /tmp/test_disk.img /tmp/evidence.img
   ```
2. Document chain of custody — who handled evidence, when, and why
3. Capture volatile data in priority order:
   - Running processes: `ps auxf > /tmp/evidence/processes.txt`
   - Network connections: `ss -tulnp > /tmp/evidence/connections.txt`
   - Logged-in users: `w > /tmp/evidence/users.txt`
   - System uptime and time: `date && uptime > /tmp/evidence/timing.txt`
4. Preserve log files before they rotate:
   ```bash
   cp /var/log/auth.log ~/grimoire/ir/evidence/
   cp /var/log/syslog ~/grimoire/ir/evidence/
   ```
5. Hash all evidence files for integrity verification

### Phase 3: Simulated Incident Response

1. **Scenario:** An IDS alert shows outbound connections to a known malicious IP from a workstation. The SOC confirms suspicious process activity.
2. **Identify:** Review the alert details — source host, destination IP, timestamps, protocol
3. **Contain:**
   - Isolate the affected host from the network (simulate with firewall rules)
   ```bash
   # Simulate network isolation
   sudo iptables -A OUTPUT -d <malicious_ip> -j DROP
   sudo iptables -A INPUT -s <malicious_ip> -j DROP
   ```
   - Preserve running state before shutting down any processes
4. **Eradicate:** Identify the malicious process, trace its origin, remove persistence mechanisms
   ```bash
   # Find the suspicious process
   ps aux | grep -i suspicious
   # Check for persistence
   crontab -l
   ls -la /etc/cron.d/
   systemctl list-unit-files --state=enabled
   ```
5. **Recover:** Verify the host is clean, restore from backup if needed, monitor for recurrence
6. **Document:** Write a complete incident report covering all phases, timeline, and findings
